# iVDrive Release Notes: v1.0.20 🚀

## Overview
Release `v1.0.20` is a major administration update focused on giving platform owners deep visibility into fleet metrics and platform health, alongside several critical backend stability fixes. We are officially paving the way towards a fully rock-solid stable release!

## Key Features

### 📊 Admin Platform Statistics Dashboard
Administrators now have a dedicated "**Statistics**" tab in the Admin Console (`/admin`). This new overview provides anonymized, real-time analytics for the entire iVDrive ecosystem:
- **Global Reach & Fleet Distribution**: See exactly how many users and vehicles are active, broken down into horizontal bar charts showing the most popular regions (complete with beautifully injected country flags like 🇱🇹, 🇬🇧, 🇫🇷).
- **Connector Health & Sync Error Rates**: A dedicated health dashboard showing the ratio of healthy active vehicles to those stuck in `token_error` or `auth_failed` states. Instantly gauge the global Sync Error Rate.
- **Global Telemetry Metrics**: Huge rolling numbers showing exactly how many Trips and Charging Sessions iVDrive has accurately captured to date.
- **Model Distribution**: A clean visual breakdown of which specific Skoda Enyaq models (e.g. iV 80, RS) make up the connected fleet.

## Critical Bug Fixes

### 🔌 Target SoC Accuracy for Parked Vehicles
We identified a subtle bug in our backend "Smart Polling" system: when a vehicle was disconnected and completely parked, the collector would skip saving updated `charging_settings` to conserve database space. This inadvertently discarded user-initiated changes to the **Target State of Charge (SoC)**, causing the web dashboard to occasionally show an outdated Target limit (e.g. stuck at 80% when set to 90%).
- **The Fix**: The Smart Polling loop now actively watches for `Target SoC` changes, persisting them directly to the `charging_states` table even if the vehicle is deeply asleep or disconnected. 

### 🗄️ Database Migration Splitting
Some backend deployments experienced crashes during container upgrades due to an `Alembic` error complaining about "Multiple head revisions" (a split timeline in our database schema history following the recent indexing updates).
- **The Fix**: Manually repaired the migration revision chain (`f3f2d2b1c4c9_add_missing_indexes.py`) directly pointing back to the correct parent `709fd0a72bf`. Database upgrades (`alembic upgrade head`) will now flow seamlessly.
